Restore Instructions ▼

{# ── Option A ── #}

Option A — Config snapshot restore

Use this when the Pi hardware is intact but the OS is corrupted, or when migrating to a new Pi. Requires a fresh Pi OS install first.

  1. Flash a fresh Raspberry Pi OS Lite (64-bit) to a new NVMe using Raspberry Pi Imager.
  2. Boot and run the BeaconButty setup scripts to reinstall dependencies (Zeek, ClickHouse, RITA, Suricata).
  3. Copy your snapshot file to the Pi, then extract it over /: 
    sudo tar -xzf config-YYYY-MM-DD.tar.gz -C /
  4. Restore the package list: 
    sudo dpkg --set-selections < packages-YYYY-MM-DD.txt
sudo apt-get -y dselect-upgrade
  5. Reload systemd and re-enable services: 
    sudo systemctl daemon-reload
sudo systemctl enable --now \
  zeek.service bb-graphs.service bb-watchdog.service bb0-display.service \
  beaconbutty-assets.timer beaconbutty-backup.timer beaconbutty-housekeeping.timer \
  beacon-report.timer rita-analyze.timer suricata-update.timer wan-watchdog.timer \
  iptables.service ip6tables.service log2ram.service log2ram-daily.timer
  6. Reboot.

Note: ClickHouse beacon history is not included in the snapshot. Historical data will be lost but the system will start collecting fresh data immediately.

{# ── Option B ── #}

Option B — Full archive restore

Use this when you want to recover the entire system — configs, ClickHouse history, logs — without a bootable clone. Requires a fresh Pi OS Lite install first; the archive is extracted over the top of it.

  1. Flash a fresh Raspberry Pi OS Lite (64-bit) to a new NVMe using Raspberry Pi Imager. Boot it and create the dm user as before.
  2. Copy archive-YYYY-MM-DD.tar.gz onto the Pi (or mount the USB drive it's stored on).
  3. Stop services that hold files open and extract the archive over /: 
    sudo systemctl stop clickhouse-server zeek suricata bb-graphs || true
sudo tar -xzpf archive-YYYY-MM-DD.tar.gz -C / --numeric-owner --xattrs
  4. Reload systemd and reboot: 
    sudo systemctl daemon-reload
sudo reboot

ClickHouse history IS preserved. Package versions are carried in the archive as installed binaries; if the fresh Pi OS has newer versions, extraction may overwrite them — run sudo apt update && sudo apt -f install if services fail to start.

{# ── Option C ── #}

Option C — Full-disk USB clone restore

Use this for a complete bare-metal restore. Recovers everything including ClickHouse history. The USB clone is itself a bootable Pi OS — boot from it first, then clone back to a new NVMe.

Scenario A — NVMe failed, replacing it

  1. Power off bb0. Remove the failed NVMe. Insert the USB clone stick.
  2. Configure the Pi to boot from USB. On a Pi 5, USB boot should already be in the default boot order after NVMe. If not: 
    sudo rpi-eeprom-config --edit
# Set: BOOT_ORDER=0xf416  (USB before NVMe, NVMe before SD)
  3. Power on — the Pi will boot from the USB clone.
  4. Insert the new NVMe (via USB adapter or direct slot).
  5. Clone from USB back to the new NVMe: 
    sudo rpi-clone nvme0n1
  6. Power off. Remove the USB stick. Reboot — Pi boots from restored NVMe.

Scenario B — Migrating to a new Pi 5

  1. Insert the USB clone into the new Pi 5 (no NVMe installed).
  2. Boot the new Pi from USB.
  3. Insert the new NVMe (via adapter or directly).
  4. Clone USB → new NVMe: 
    sudo rpi-clone nvme0n1
  5. Power off. Remove USB. Boot from NVMe.
Pi 5 boot order reference
BOOT_ORDER=0xf416 — USB → NVMe → SD → restart
BOOT_ORDER=0xf461 — NVMe → USB → SD → restart (default after NVMe install)
Check current order: sudo rpi-eeprom-config | grep BOOT_ORDER
